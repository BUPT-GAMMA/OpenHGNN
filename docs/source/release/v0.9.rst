OpenHGNN v0.9 Release Notes
===========================

OpenHGNN v0.9 focuses on model coverage, reproducibility, and documentation
standardization. The release adds 10 model contributions, introduces
``node_regression``, updates the recommended environment, and documents the
minimum reproduction standard for model results.

Release Highlights
------------------

.. list-table::
   :header-rows: 1

   * - Area
     - Update
     - User impact
   * - Models
     - 10 new model contributions and 11 registered model names
     - More recent heterogeneous graph methods are available through the
       OpenHGNN registry.
   * - Tasks
     - New ``node_regression`` task
     - Continuous node-label prediction can use the same public
       ``main.py`` and ``Experiment(...)`` entry points as classification and
       link prediction.
   * - Environment
     - v0.9 pinned environment with Python 3.11, PyTorch 2.4.0, DGL
       2.4.0+cu121, and CUDA 12.1
     - Reproduction and CI can use the same dependency files.
   * - Documentation
     - New model overview, task overview, reproduction guide, and model PR
       checklist
     - Users and contributors have a clearer path from installation to model
       reproduction and review.
   * - Validation
     - Registry validation, PR hygiene checks, and changed-model smoke-test
       selection
     - Release candidates are checked against OpenHGNN integration and
       repository hygiene requirements.

What Still Needs Attention
--------------------------

The v0.9 model batch is merged for release coverage, but several pieces should
continue to improve after release:

* The ten v0.9 model reproduction notes have been moved into
  :doc:`../models/index`; historical model-specific READMEs under legacy
  ``openhgnn/output/<MODEL>/README.md`` paths should be migrated in later
  cleanup passes.
* Some model reproduction entries still depend on model-specific datasets or
  external preprocessing steps that should be converted into stable scripts.
* HGSketch currently has component-level validation only because OpenHGNN does
  not yet expose a general ``graph_classification`` task registry entry.
* RelGT depends on RelBench data preparation, sentence-transformers, and cached
  table features; keep its full benchmark outside default CI until a small
  fixture is available.
* Full paper-level reproduction should report hardware, seed count, mean/std,
  command line, and log path using the standard
  :doc:`../get_started/reproduce_model` format.
